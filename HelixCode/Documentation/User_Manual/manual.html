#Overview

HelixCode is a distributed, AI-powered software development platform with multi-platform support. It revolutionizes software development by combining the power of distributed computing with advanced AI capabilities, enabling developers to build better software faster.

🎯 What Makes HelixCode Unique?

HelixCode is the world's first distributed AI development platform that combines:

  • 9 AI provider integrations including Anthropic Claude and Google Gemini
  • Extended thinking capabilities for complex problem-solving
  • Massive 2M token context windows for entire codebase analysis
  • Multi-client support (CLI, TUI, REST API, Mobile)
  • Enterprise-grade security and compliance

#Features

Core Features

  • Multi-Platform Support: Desktop, mobile, terminal, and specialized OS clients
  • Distributed Computing: Worker nodes for parallel task execution
  • AI Integration: LLM-powered code generation and reasoning with multiple free providers
  • Free AI Models: Access to XAI (Grok), OpenRouter, GitHub Copilot, and Qwen without API keys
  • Real-time Collaboration: MCP protocol for tool execution
  • Authentication & Security: JWT-based auth with session management
  • Task Management: Checkpoint-based work preservation
  • Notification System: Multi-channel notifications (Slack, Email, Discord)

#Production Deployment

1. Clone and Setup

git clone https://github.com/HelixDevelopment/Helix-CLI.git
cd helixcode
cp .env.example .env

2. Configure Environment Variables

Edit the .env file with your production values:

HELIX_AUTH_JWT_SECRET=your-super-secure-jwt-secret
HELIX_DATABASE_PASSWORD=your-secure-database-password
HELIX_REDIS_PASSWORD=your-secure-redis-password

3. Deploy with Docker Compose

docker-compose up -d

4. Check Deployment

docker-compose ps
curl http://localhost/health

#Architecture

#Core Components

  • Server: Main API server with REST and WebSocket endpoints
  • Database: PostgreSQL for persistent data storage
  • Cache: Redis for session and task state management
  • Workers: Distributed worker nodes for task execution
  • MCP Server: Model Context Protocol for AI tool integration

#CLI User Guide

The HelixCode CLI provides a powerful command-line interface for managing workers, tasks, LLM interactions, and notifications.

#Installation

Building from Source

# Navigate to the HelixCode directory
cd HelixCode

# Build the CLI
go build -o bin/helix-cli ./cmd/cli/

# Add to your PATH (optional)
export PATH="$PATH:$(pwd)/bin"

# Verify installation
helix-cli --help

Pre-built Binaries

# Download from releases
curl -L https://github.com/helixcode/helixcode/releases/latest/download/helix-cli-linux-amd64 -o helix-cli
chmod +x helix-cli
sudo mv helix-cli /usr/local/bin/

#Complete Command Reference

--key SSH private key path --key ~/.ssh/worker_key

Usage Examples

#Server Administration Guide

This guide covers deploying, configuring, and monitoring the HelixCode server in production environments.

#Deployment

Docker Compose (Recommended)

# docker-compose.yml
version: '3.8'

services:
  helixcode:
    image: helixcode/server:latest
    ports:
      - "8080:8080"
    environment:
      - HELIX_AUTH_JWT_SECRET=${JWT_SECRET}
      - HELIX_DATABASE_HOST=postgres
      - HELIX_DATABASE_PASSWORD=${DB_PASSWORD}
      - HELIX_REDIS_HOST=redis
      - HELIX_REDIS_PASSWORD=${REDIS_PASSWORD}
    depends_on:
      - postgres
      - redis
    restart: unless-stopped

  postgres:
    image: postgres:15
    environment:
      - POSTGRES_USER=helix
      - POSTGRES_PASSWORD=${DB_PASSWORD}
      - POSTGRES_DB=helixcode_prod
    volumes:
      - postgres_data:/var/lib/postgresql/data
    restart: unless-stopped

  redis:
    image: redis:7-alpine
    command: redis-server --requirepass ${REDIS_PASSWORD}
    volumes:
      - redis_data:/data
    restart: unless-stopped

  nginx:
    image: nginx:alpine
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf:ro
      - ./certs:/etc/nginx/certs:ro
    depends_on:
      - helixcode
    restart: unless-stopped

volumes:
  postgres_data:
  redis_data:

#Configuration

Configuration File Structure

# config/config.yaml

server:
  address: "0.0.0.0"
  port: 8080
  read_timeout: 30      # seconds
  write_timeout: 30     # seconds
  idle_timeout: 300     # seconds
  shutdown_timeout: 30  # graceful shutdown

database:
  host: "${HELIX_DATABASE_HOST:localhost}"
  port: 5432
  user: "${HELIX_DATABASE_USER:helix}"
  password: "${HELIX_DATABASE_PASSWORD}"  # REQUIRED
  dbname: "${HELIX_DATABASE_NAME:helixcode_prod}"
  sslmode: "disable"    # Use "require" in production

redis:
  host: "${HELIX_REDIS_HOST:localhost}"
  port: 6379
  password: "${HELIX_REDIS_PASSWORD}"
  db: 0
  enabled: true

auth:
  jwt_secret: "${HELIX_AUTH_JWT_SECRET}"  # REQUIRED - 32+ chars
  token_expiry: 86400      # 24 hours
  session_expiry: 604800   # 7 days
  bcrypt_cost: 12

workers:
  health_check_interval: 30   # seconds
  health_ttl: 120             # seconds
  max_concurrent_tasks: 10

tasks:
  max_retries: 3
  checkpoint_interval: 300    # 5 minutes
  cleanup_interval: 3600      # 1 hour

llm:
  default_provider: "local"
  max_tokens: 4096
  temperature: 0.7
  timeout: 30
  max_retries: 3

  selection:
    strategy: "performance"   # performance, cost, availability, round-robin
    fallback_enabled: true
    health_check_interval: 30

logging:
  level: "info"      # debug, info, warn, error
  format: "text"     # text, json
  output: "stdout"   # stdout, file path

#Workflow Authoring Guide

Workflows define automated sequences of steps for development tasks. This guide explains how to create and configure workflows.

#Core Concepts

Workflow Structure

// Workflow represents a development workflow
type Workflow struct {
    ID          string         // Unique identifier
    Name        string         // Human-readable name
    Description string         // Workflow description
    Mode        string         // Workflow mode
    Steps       []Step         // Ordered steps
    Status      WorkflowStatus // Current status
}

// WorkflowStatus values:
// - pending    : Not yet started
// - running    : Currently executing
// - completed  : Successfully finished
// - failed     : Execution failed

Creating a Workflow

# workflow.yaml
id: "build-and-test-workflow"
name: "Build and Test"
description: "Compile code, run linter, execute tests"
mode: "sequential"

steps:
  - id: "analyze"
    name: "Code Analysis"
    type: "analysis"
    action: "analyze_code"

  - id: "lint"
    name: "Lint Code"
    type: "validation"
    action: "lint_code"
    dependencies: ["analyze"]

  - id: "build"
    name: "Build Project"
    type: "execution"
    action: "build_project"
    dependencies: ["lint"]

  - id: "test"
    name: "Run Tests"
    type: "execution"
    action: "run_tests"
    dependencies: ["build"]

#Step Types

Type Purpose Example Actions
analysis Analyze code structure, patterns, dependencies analyze_code
generation Generate code, documentation, tests generate_code
execution Run commands, scripts, builds execute_command, run_tests, build_project
validation Validate code quality, security, compliance lint_code

Step Actions Reference

Action Description
analyze_code Perform static code analysis
generate_code Generate code using LLM
execute_command Run arbitrary shell command
run_tests Execute test suite
lint_code Run linter (golangci-lint, eslint, etc.)
build_project Compile/build the project

#Dependencies

DAG-Based Execution

Steps form a Directed Acyclic Graph (DAG). Steps without dependencies run first; dependent steps wait for their prerequisites.

# Complex dependency example
steps:
  - id: "init"
    name: "Initialize"
    type: "execution"
    action: "execute_command"
    # No dependencies - runs first

  - id: "analyze-frontend"
    name: "Analyze Frontend"
    type: "analysis"
    action: "analyze_code"
    dependencies: ["init"]

  - id: "analyze-backend"
    name: "Analyze Backend"
    type: "analysis"
    action: "analyze_code"
    dependencies: ["init"]
    # Runs in parallel with analyze-frontend

  - id: "generate-types"
    name: "Generate Types"
    type: "generation"
    action: "generate_code"
    dependencies: ["analyze-frontend", "analyze-backend"]
    # Waits for both analyses to complete

  - id: "build"
    name: "Build All"
    type: "execution"
    action: "build_project"
    dependencies: ["generate-types"]

Step Status Flow

pending -> running -> completed
                   \-> failed
                   \-> skipped (if dependency failed)
