--------------------------------------------------------------------------------
READING NOTES / OBSERVATIONS (for Phase 1+ task authors)
--------------------------------------------------------------------------------
* CLI startup — NewCLI() costs ~1.77 ms and ~4 462 allocations cold. This is
  the number P1-T02 (lazy Ollama discovery) and P1-T03 (sync.Once lazy startup)
  must move. NOTE: a parallel variant of this benchmark was trialled and
  EXPOSED A REAL CONCURRENCY DEFECT — NewCLI() -> config.Load() ->
  process-global viper.SetDefault is not goroutine-safe and concurrent
  construction panics with "concurrent map writes" (viper.go:1492). NewCLI() is
  constructed once per process, so this is not a supported usage pattern; the
  parallel benchmark is deliberately omitted because it would only re-trigger a
  panic that cannot be fixed without a production-code change (out of P0-T02
  scope). Flagged here for the speed programme — Phase 1/Phase 4 may want a
  fix-and-paired-benchmark follow-up.

* LLM dispatch — RequestBuild (request conversion only) is cheap at ~161 ns /
  2 allocs. The full RoundTrip against the local httptest server is ~82.5 us /
  127 allocs — this is HelixCode's own per-call overhead (JSON marshal/unmarshal
  + http.Client dispatch + response conversion), NOT network latency. P1-T01
  (shared tuned HTTP/2 transport) targets the RoundTripParallel number, which is
  already ~2x faster per-op than serial here purely from CPU parallelism on the
  zero-latency local server; the real win on a remote provider is removing the
  per-call TLS handshake on bursts, which this local-server benchmark cannot
  show — P1-T01 must additionally capture netstat/idle-conn evidence.

* Repo-map build — ~16.7 ms over a 30-file synthesized Go tree, ~19 485 allocs.
  This is the serial-parse cost (B04). P2-T03 (content-addressed cache),
  P2-T04 (worker pool + parser pool + single-pass stats) and P2-T06
  (incremental parse) all target this number. OptimalContext is slightly higher
  (~17.3 ms) because it adds ranking + context assembly on top of the build.

* Content search — ~717 us literal / ~968 us regex over a 60-file synthesized
  text tree, ~1 820 / ~1 122 allocs. This is the serial walk+read cost (B08)
  that P2-T05 (bounded worker-pool parallel search) must move.

Re-run this exact command after each Phase 1+ task and paste the new numbers
alongside these to prove the speedup (CONST-035 / Rule 9). benchstat over
multiple -count runs is recommended for statistically-sound deltas.
