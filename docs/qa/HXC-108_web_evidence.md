## Harness (TCC-free, headless Chrome)

`helix_code/tmp/web_record/main.go` (gitignored `tmp/`; `go vet ./tmp/web_record/` → clean). It:
1. launches headless Chrome (`ExecPath` = the host Google Chrome, 1280×1400 viewport),
2. navigates to `http://localhost:18080/`,
3. drives the REAL user journey through the page's own UI — fill the real login form → click **Sign in** (mints a real server-side JWT) → fill provider/model/prompt → click **Generate**,
4. captures one PNG frame per step from the browser viewport,
5. reads the live DOM at each step and **asserts real rendered content** (a 401 / error page / empty output is a FAIL, never a faked PASS — §11.4.2/§11.4.107).

Credentials are runtime-generated by the caller and passed via env (`WEB_USER`/`WEB_PASS`) — **never hardcoded, never logged into evidence** (§11.4.10).

## Real upstream finding (honest, §11.4.6 — not a bluff, surfaced for the conductor)

The web Generate form with an EMPTY model returned **HTTP 502**:
```
generation failed: DeepSeek request failed: DeepSeek API returned status 400:
  {"error":{"message":"The supported API model names are deepseek-v4-pro or
   deepseek-v4-flash, but you passed ."}}
```
This is a **real upstream API change**, not a transient: DeepSeek deprecated `deepseek-chat` (which `/api/v1/llm/providers` still advertises for the `deepseek` provider) in favour of `deepseek-v4-pro` / `deepseek-v4-flash`. With an explicit valid model (`deepseek-v4-flash` or `deepseek-v4-pro`) the same endpoint returns a real `content:"4"` (verified both via curl AND via the web UI above). The recording therefore drives the journey with an explicit valid model. **Suggested follow-up (conductor):** the server's default-model resolution for DeepSeek (empty model → request body `model=""`) needs to map to a currently-supported model, and the `/llm/providers` catalog for `deepseek` should reflect `deepseek-v4-*` rather than the deprecated `deepseek-chat`.
