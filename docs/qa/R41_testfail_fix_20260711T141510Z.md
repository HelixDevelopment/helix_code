## Failure 1 — `internal/security` `TestRawText_EmittedByDefault`

### Root cause (investigated, cited)

`helix_code/internal/security/translator.go` `init()` (commit `1254e0a6`, 2026-06-28,
"HXC-097 §11.4 anti-bluff, 2026-06-15") changed the package-level `defaultTranslator`
from a hardcoded `securityi18n.NoopTranslator{}` (raw message-ID echo) to the **real
embedded-bundle translator** (`securityi18n.NewTranslator()`), whenever the `go:embed`'d
`active.en.yaml` bundle loads successfully — which it always does in this test binary,
since `go:embed` content is compiled in and does not depend on the working directory or
any runtime filesystem state. `translator.go`'s own doc comment states the intent
explicitly:

> "SetTranslator(nil) restores THIS — 'nil = restore default' means 'restore correct
> prose', not 'revert to raw-key echo'."

This is a **deliberate, already-shipped anti-bluff fix**: library code previously emitted
raw message-ID keys on any entry path that never called `i18nwiring.WireAll()`. The SAME
commit (`1254e0a6`) reconciled two sibling tests
(`TestTr_DefaultsToNoopTranslator`, `TestSetTranslator_NilResetsToNoop`) to the new
behavior, but **missed** `TestRawText_EmittedByDefault`, which was left asserting the
stale pre-HXC-097 contract ("with no translator wired, tr() returns the raw ID").

Confirmed empirically (Phase 1 evidence-gathering) BEFORE the fix:

```
=== RUN   TestRawText_EmittedByDefault
2026/07/11 19:02:27 [INFO] Starting security scan for feature: noop-emit
2026/07/11 19:02:27 [INFO] No security scanners available for: noop-emit
2026/07/11 19:02:27 [INFO] Set SONARQUBE_URL/SONARQUBE_TOKEN or SNYK_TOKEN to enable security scanning
    translator_test.go:190: rec = "No security scanners configured", want contain raw ID (Noop echo)
--- FAIL: TestRawText_EmittedByDefault (0.00s)
```

`"No security scanners configured"` is the real bundle prose for
`internal_security_recommendation_no_scanners` (`active.en.yaml:45-46`) — proving the
real translator, not Noop, is genuinely the active default. This is NOT a regression the
test caught; it is the OLD test asserting behavior the codebase intentionally changed.

### Decision: RECONCILE (§11.4.120), not fake-pass, not code revert

Per §11.4.120: the correct response is neither (1) fake-passing the gate (weakening the
assertion to a tautology / deleting it) nor (2) reverting the correct HXC-097 fix to
satisfy the stale test — it is to rewrite the test to assert the NEW genuine mechanism,
backed by captured evidence of the new correct behavior.

`helix_code/internal/security/translator_test.go` `TestRawText_EmittedByDefault` was
rewritten to:

- Assert the resolved bundle prose (`"No security scanners configured"`) is what's
  actually emitted with no translator explicitly wired (only the implicit nil-reset via
  `resetTranslator(t)`).
- ADD a reverse guard: assert the raw message ID does **not** leak into the output — so
  the test now also catches a future regression back to `NoopTranslator{}`-by-default
  (closing the loop both directions, matching the §1.1 paired-mutation spirit).
- Carry a full citation of the root cause (commit, HXC-097 tag, §11.4.120) in the doc
  comment, per the "reconciliation MUST be a visible, evidence-cited change" mandate.

## Failure 2 — `internal/tools/browser` `TestBrowserActions`

### Root cause (investigated, cited — real bug, NOT flakiness)

Phase 1 evidence-gathering: ran the test BEFORE any fix with full output captured:

```
=== RUN   TestBrowserActions
=== RUN   TestBrowserActions/navigate_to_url
    browser_test.go:213: Navigated to https://example.com/ - Example Domain
=== RUN   TestBrowserActions/get_page_info
=== RUN   TestBrowserActions/evaluate_javascript
    browser_test.go:236: Error Trace: ... Expected value not to be nil.
    browser_test.go:237: JavaScript result: <nil> (type: undefined)
=== RUN   TestBrowserActions/get_element
    browser_test.go:242: Error: Received unexpected error: context deadline exceeded
=== RUN   TestBrowserActions/get_multiple_elements
    browser_test.go:259: Error: Received unexpected error: context deadline exceeded
=== RUN   TestBrowserActions/scroll_page
    browser_test.go:274: Error: Received unexpected error: context deadline exceeded
--- FAIL: TestBrowserActions (31.35s)
    --- PASS: TestBrowserActions/navigate_to_url (26.56s)
    --- PASS: TestBrowserActions/get_page_info (2.02s)
    --- FAIL: TestBrowserActions/evaluate_javascript (2.02s)
    --- FAIL: TestBrowserActions/get_element (0.00s)
    --- FAIL: TestBrowserActions/get_multiple_elements (0.00s)
    --- FAIL: TestBrowserActions/scroll_page (0.00s)
```

Evidence gathered to rule out environment-unavailability (§11.4.3 SKIP path):

- Chromium binary present: `/usr/bin/chromium`, `/usr/bin/chromium-browser`.
- Network reachability confirmed: `curl https://example.com` → `HTTP 200` in `0.29s`.
- Raw headless-chromium benchmark, **entirely outside Go/chromedp**, to characterize this
  host's genuine chromium cold-start cost:
  ```
  $ time chromium --headless --disable-gpu --no-sandbox --disable-dev-shm-usage \
        --dump-dom https://example.com
  elapsed: 25.27s
  ```
  This host's headless-Chrome cold start (process spawn + first render pipeline) is
  genuinely ~25s — confirmed independent of any Go code, ruling out a chromedp-specific
  or test-specific slowness.

Root-cause trace (Phase 2/3, comparison against sibling `ActionExecutor` methods +
`Controller.Launch`):

`helix_code/internal/tools/browser/controller.go` `Launch()` derived the shared
`browserCtx` — the SAME context every subsequent action (`Navigate`, `Evaluate`,
`GetElement`, `Scroll`, ...) uses via `GetContext(browserID)` — via:

```go
if opts.Timeout > 0 {
    browserCtx, browserCancel = context.WithTimeout(allocCtx, opts.Timeout)
} else { ... }
```

`opts.Timeout` defaults to `30 * time.Second` (`DefaultLaunchOptions()`). This applied
**unconditionally**, silently overriding/shrinking the caller's own, more generous
context deadline (the test's own `180*time.Second` `ctx`, whose comment explicitly
states "180s covers worst-case launch + 6 navigate-and-act subtests..." — an assumption
the code silently violated). Combined with the ~25s real chromium cold-start cost paid
during the first `Navigate()` call, the 30s session-lifetime budget was exhausted after
~1.5 subtests: subtest 3 (`evaluate_javascript`) straddled the deadline (its
`chromedp.Run` failed with `context deadline exceeded`, but `Evaluate()` had a SECOND,
independent bug — see below — that swallowed the error into `EvaluateResult.Error`
without returning it, so the test only observed a mysteriously-nil `result.Value`), and
every subtest after that failed instantly with `context deadline exceeded` because
`browserCtx` was already permanently `Done()`.

This is a **deterministic, reproducible architecture defect** — not timing-dependent
flakiness in the classic racy sense — given this host's chromium cold-start cost. Two
real bugs, confirmed via reading + comparing against sibling code (Phase 2 pattern
analysis):

1. **`Launch()`**: `opts.Timeout`'s default value (30s) applies as a hard session-wide
   deadline regardless of the caller's own `ctx` deadline, silently shrinking a more
   generous caller budget. Zero external callers outside `internal/tools/browser/`
   exist (verified via codebase-wide grep), so this is a self-contained, in-scope fix.
2. **`Evaluate()`**: the function's own `error` return was **always `nil`** regardless of
   the underlying `chromedp.Run` outcome — the real failure was stashed only in
   `EvaluateResult.Error`, unlike every sibling `ActionExecutor` method (`Navigate`,
   `Click`, `Type`, `Scroll`, `GetElement`, `GetElements`, `GetPageInfo`), all of which
   return the real error via the standard `(T, error)` idiom. This masked the genuine
   `context deadline exceeded` failure as an inexplicable nil value. Verified zero
   external callers of `Evaluate` outside `internal/tools/browser/browser.go` (a pure
   passthrough), so fixing the signature's real-error-return has zero blast radius
   beyond this package.

### Decision: FIX (real bug), not honest SKIP

Chromium is genuinely available and usable in this environment (just slow to cold-start)
— an honest SKIP-with-reason would have been the wrong call here since the defect is a
real, fixable production-code bug, not environment unavailability.

### Fix applied

`helix_code/internal/tools/browser/controller.go` — `Launch()`: `opts.Timeout` now
applies **only** as a fallback safety-net when the caller's `ctx` carries no deadline of
its own. When `ctx` already has a deadline, that deadline is authoritative (it
propagates through `allocCtx` — `chromedp.NewExecAllocator` wraps
`context.WithCancel(ctx)`, confirmed by reading
`chromedp@v0.15.1/allocate.go:NewExecAllocator`, which preserves ancestor deadlines —
into `browserCtx` via `context.WithCancel`) and is never silently shrunk by the
`LaunchOptions` default.

`helix_code/internal/tools/browser/actions.go` — `Evaluate()`: now returns the real
`chromedp.Run` error as its own `error` return (in addition to the unchanged
`EvaluateResult.Error` field), matching every sibling `ActionExecutor` method's contract.
