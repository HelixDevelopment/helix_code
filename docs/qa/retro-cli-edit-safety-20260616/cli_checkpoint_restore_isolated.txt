in the SAME isolated repo:

      create -> bad edit -> restore  ==> CORRECTLY restored ("TRACKED good v1" returned). WORKS.

        "✅ working tree restored to checkpoint <id>"
REAL DEFECT (anti-bluff finding): the git-backend /checkpoint snapshots & restores only
git-TRACKED content, but /checkpoint restore prints an unconditional success message even
when untracked working-tree files are NOT restored. A user who checkpoints a working tree
containing untracked files, then edits/corrupts them, then restores, gets a green
"working tree restored" message while those untracked files are silently left corrupted.
This is a §11.4 sink-side success-message-vs-reality mismatch for the untracked-file case.
Classification: real current defect, bounded to untracked files; tracked-file restore is
genuinely working. Reported to the conductor for tracker filing (NOT fixed here — source is
out of this stream's scope).
