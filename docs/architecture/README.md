This folder documents the **model-access + provider-exposure + CLI-bridge** system: how a
recognized API key turns into the WORKING models a user can use, how every provider/ensemble/
HelixLLM/CLI-agent is exposed under one uniformly-named root, and how the CLI-bridge proxies
prompts through real CLI binaries.

## Honesty note (§11.4 / §11.4.123)

Both documents describe the **target architecture**. They explicitly separate **IMPLEMENTED**
components (real code, cited `file:line`) from **PLANNED** components (specified in SP1/SP2/SP4
but not yet built — no captured runtime evidence). A PLANNED node is a design intent, not a claim
of working software. The runtime-evidence bar (§11.4.5 / §11.4.69 / §11.4.107) is met only by the
tests/Challenges each SP plan requires at execution time — not by these diagrams or this DDL.

Quick orientation on current state (from the analysis passes):

- **IMPLEMENTED today:** real LLMsVerifier REST consumption (`verifier/client.go`,
  `adapter.go:183-224`); viper key binding (`config.go:307-345`); the CLI-exec primitive
  (`base.go:120-138`); config re-export + static `Validate()` (`unified_cli_generator.go`).
- **DEAD/UNWIRED (wire, don't delete — §11.4.124):** `secrets.LoadAPIKeys` (`loader.go:30`).
- **Known bluffs to remediate:** CLI shows failed/pending models (D-2), `/v1/completion/models`
  hardcoded 3-model list (D-1), hardcoded provider lists (D-5/CONST-036), `qwencode`/
  `instance_manager` CLI stubs (D-6).
- **PLANNED (not built):** the `GetWorkingModels` filter, key-presence gate, unified
  `GET /v1/catalog`, HelixLLM root promotion, the `CLIAgentProvider` adapter, `replace
  dev.helix.agent` wiring (D-7), filesystem INSTALL + LIVE post-install validation.
