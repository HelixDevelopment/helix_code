3. SWAP procedure (Task 3) — .gitmodules rewrite + sync + per-fork upstream remote

OPERATOR-CONFIRM at execution. Swapping 49 submodule URLs is high-blast-radius (§11.4.101: irreversible-ish in effect on every developer clone, high blast radius). Gate the first mutating run behind an explicit operator yes. Default to a plan/dry-run.

3.1 Pre-flight (read-only, §11.4.71 fetch-first)

cd /Volumes/T7/Projects/HelixCode
git fetch --all --prune --tags
git submodule status cli_agents/        # capture pre-state SHAs (evidence baseline)
awk '!seen[$0]++' docs/caf/map.tsv > /tmp/map_uniq.tsv   # dedup defensively (49 rows)
test "$(wc -l < /tmp/map_uniq.tsv)" -eq 49 || { echo "map not 49 unique"; exit 1; }

3.2 .gitmodules URL rewrite (per row)

For each of the 49 unique rows, rewrite ONLY submodule.cli_agents/<name>.url to the GitHub fork column. claude-code-source is never in the map, so it is never touched.

while IFS=$'\t' read -r name up gh gl; do
  [ -z "$name" ] && continue
  case "$name" in \#*) continue ;; esac
  git config -f .gitmodules "submodule.cli_agents/$name.url" "$gh"
done < /tmp/map_uniq.tsv

3.3 Sync the config into .git/config + each worktree, then update

git submodule sync --recursive          # push .gitmodules urls into .git/config + submodule .git/config
git submodule update --init --recursive cli_agents/   # re-point each working submodule at the fork

Because the fork is a true fork of upstream, the previously-checked-out commit SHA is present in the fork’s object graph, so git submodule update re-points cleanly without re-clone in most cases. If a pinned SHA is NOT yet in the fork (fork behind upstream), update that fork first via update_fork_from_upstream.sh --only <name> --execute (P3.4 machinery) THEN re-run git submodule update for it.

3.4 Keep upstream as a named remote per fork (§11.4.71 merge-from-upstream)

Inside each submodule working tree, origin now = the caf-* fork. Add upstream as a second named remote so future update_fork_from_upstream.sh (and manual merges) can fetch upstream:

while IFS=$'\t' read -r name up gh gl; do
  d="cli_agents/$name"
  ( cd "$d" 2>/dev/null \
    && git remote get-url origin >/dev/null 2>&1 \
    && { git remote add upstream "$up" 2>/dev/null || git remote set-url upstream "$up"; } \
    && git remote add gitlab "$gl" 2>/dev/null || git remote set-url gitlab "$gl" )
done < /tmp/map_uniq.tsv

Result per submodule: origin→ GitHub fork (fetch+push), upstream→ third-party source (fetch only, for §11.4.71 merges), gitlab→ GitLab mirror (push). NOTE: the upstream/gitlab remotes live in each submodule’s .git/config (per-clone, NOT versioned in .gitmodules); document them in docs/caf/README.md so a fresh clone re-adds them (or wire into scripts/init-submodules.sh).

3.5 Buildability spot-check (anti-bluff, §11.4.5 captured evidence)

The cli_agents submodules are reference agents (multi-language), not part of the Go build graph, so “buildable” = (a) submodule checks out at its pinned SHA, (b) a representative few build/parse in their own toolchain. Spot-check ≥3 across languages:

git submodule status cli_agents/ | grep -v '^-'   # no '-' prefix => all initialised (not de-init)
# Rust:   ( cd cli_agents/aichat   && cargo metadata --no-deps >/dev/null )   # crush is Go; aichat is Rust
# Go:     ( cd cli_agents/crush    && go vet ./... 2>&1 | head )
# Python: ( cd cli_agents/aider    && python -c 'import sys; print(sys.version)' && ls pyproject.toml )

Capture stdout/exit codes as the evidence artefact. A swap that leaves a submodule at a SHA the fork lacks → git submodule update errors → that fork needs P3.4 sync first (see §3.3).

3.6 Rollback (reversibility — §11.4.101)

The whole swap is one .gitmodules edit + config sync. To revert before commit: git checkout .gitmodules && git submodule sync --recursive && git submodule update --init --recursive cli_agents/. Keep a copy of pre-swap git submodule status cli_agents/ as the known-good baseline.

4. GitLab-population step (Task 4) — P3.4

The 49 caf-* GitLab repos are empty_repo=True. Populate each by mirroring the fork content (the GitHub caf-* fork already holds full upstream history) into the GitLab repo — fast-forward only, NEVER force (§11.4.113).

Reuse the existing machinery (CONST-074/§11.4.74 — extend, don’t reimplement): scripts/caf/update_fork_from_upstream.sh already: clones the GitHub fork (origin), adds upstream, fetches, merges upstream main/master (ff-preferred, conflicts parked, never -s ours), then pushes to all provider remotes via caf_provider_remotes → caf_safe_push. caf_safe_push/caf_assert_no_force refuse --force/--force-with-lease/--mirror/+ref (§11.4.113-safe by construction). Providers come from CAF_PROVIDERS csv (default includes github,gitlab), so a single run also seeds GitLab.

Because the GitLab side is empty, the first push to gitlab is a fresh-branch push (still a fast-forward against an empty ref — no history to overwrite, no force needed):

# DRY-RUN (default, mutate nothing) — plan the merges + pushes:
bash scripts/caf/update_fork_from_upstream.sh --map-file /tmp/map_uniq.tsv

# OPERATOR-CONFIRM execute (G-1-push gate, §11.4.101): seeds GitHub fork (idempotent ff)
#   AND populates the empty GitLab mirror, fast-forward only:
bash scripts/caf/update_fork_from_upstream.sh --map-file /tmp/map_uniq.tsv --execute

Notes: - The script’s caf_detect_default_branch resolves main/master from the upstream symref per fork (handles mixed default branches). - Conflicts are parked to docs/caf/conflicts/<fork>_<ts>.md and the loop continues (zero-idle); none expected for an initial seed. - For forks whose pinned submodule SHA was missing (see §3.3), this run brings the fork current first, unblocking §3.3 git submodule update. - If a leaner “seed only” is preferred over merge-from-upstream, an equivalent is a per-fork git push gitlab --all && git push gitlab --tags from a fork clone — still ff-only, no --mirror (mirror is force-class and is blocked). The existing script is preferred (already anti-bluff + §11.4.113-guarded + has a Challenge).

6. Nested cline → cline-bench (handled)

cli_agents/cline/.gitmodules declares a nested submodule:

[submodule "evals/cline-bench"]
    path = evals/cline-bench
    url = https://github.com/cline/cline-bench.git

Two facts: 1. It is NOT a top-level cli_agents/ entry, so it is correctly absent from map.tsv (the map only forks the 49 top-level reference agents). 2. Its URL is HTTPS — a Rule 3 violation if/when initialized, and a CONST-051(C) nested-own-org-chain concern only if it pointed at an own-org repo (it points at third-party cline/cline-bench, so the nested-chain ban does not apply — third-party nested submodules are exempt per CONST-051(C)).

Recommendation for P3.3/P3.4 (own scope, not blocking the 49-swap): - Do not recursively init evals/cline-bench during the swap (git submodule update for cli_agents/cline should NOT pass --recursive into cline’s own nested tree, or should skip it) to avoid pulling an HTTPS submodule into the tree. - If cline-bench is needed, fork it to vasic-digital/caf-cline-bench and add it to the map as a 50th row + place it at the project root per CONST-051(C) (flat <root>/caf-cline-bench or grouped), then have cline reference it via config injection — but this is a separate, operator-gated work item, NOT part of the 49-swap. Track it as its own ticket.
